Documentation Style Analysis: spartan2_og
1. Doc Comment Style
Level	Style	Usage
Public items (fn, struct, trait, enum)	///	Universal — no /** */ block style
Module headers	//!	2–4 lines, purpose-focused
Inline (function bodies)	//	Separate lines, rarely end-of-line
Standard doc comment structure:


Summary line (one sentence)
 ↓
Extended description (context/behavior)
 ↓
# Arguments  (bulleted list)
# Returns    (description of result)
# Errors     (Result-returning fns)
# Panics     (rare, only if applicable)
2. Mathematical Notation
LaTeX is used inline and block-style in doc comments:


/// $$
/// \tilde{eq}(x, e) = \prod_{i=1}^m(e_i \cdot x_i + (1 - e_i) \cdot (1 - x_i))
/// $$
///
/// Vector $Z$ indicates $Z(e)$ where $e$ ranges from $0$ to $2^m-1$.
Inline math: $...$
Display math: $$...$$
Symbols written as Unicode (×, ·) or LaTeX inside $
3. Inline Comment Types
a) Algorithm step labels (most common):


// eval 0: a_low × b_low
// eval 2: (2·a_high - a_low) × (2·b_high - b_low)
b) Data layout explanations:


// Split A into 4 non-overlapping regions: [a_ll | a_lh | a_hl | a_hh]
// a_ll = A[0..quarter],  a_lh = A[quarter..n]
c) Constraint/validation rationale:


// we do not need to check if poly(0) + poly(1) = e, as
// decompress() call above already ensures that holds
d) Round-specific logic labels:


// Round 0: compute only (no prior challenge to fold with)
// Rounds 1+: fused fold-by-previous-r + compute in a single pass
4. Error Enum Docs
Each variant documented as /// returned if <condition>:


/// returned if the supplied witness is not of the right length
InvalidWitnessLength,

/// returned if the provided commitment is not of sufficient length
#[error("InvalidCommitmentLength: {reason}")]
InvalidCommitmentLength { reason: String },
5. Trait Documentation
Trait items use concise single-line docs (no multi-section structure):


/// A type that represents the prover's key
type ProverKey: ...;

/// Produces the keys for the prover and the verifier
fn setup<C: SpartanCircuit<E>>(...) -> Result<...>;
6. Key Observations
No doc tests — zero # Examples sections with runnable code
No usage examples in module-level //! docs
Performance notes appear in doc comments for optimized functions (e.g., explaining Montgomery reduction savings)
Inline comment density ~6% of LOC in complex modules like sumcheck.rs
Trait files favor brevity; crypto/polynomial files favor LaTeX-heavy verbose docs
Comments explain why and what, not how (trust the code for mechanics)
Quick Reference for New Code
Public fn: /// with Args/Returns/Errors sections
Module: //! 2–3 line purpose statement, list main types if multiple
Inline: Short step labels (// step name:), constraint notes (// verify that ...)
Errors: /// returned if ... on each enum variant
Math: LaTeX $...$ inline, $$...$$ for displayed formulas
Traits: One-liner per item, no sub-sections

Tone
Declarative and impersonal. No first-person ("I", "we"), no hedging ("might", "could"), no filler. The voice is the codebase speaking about itself.


/// A proof generated by the sum-check protocol.
/// Returns only the folded witness to avoid unnecessary computation on the prover side.
/// Accumulates unreduced products in wide integers, reducing only at the end.
Sentence Construction
Summary lines: gerund phrases or noun phrases, never full sentences with "this".

Pattern used	Pattern avoided
/// Verifies a sum-check proof.	/// This function verifies...
/// Fold a relaxed instance/witness...	/// This method folds...
/// Returns the degree of the polynomial.	/// This returns the degree...
Body lines: present tense, active voice, minimal words.


/// Binds the polynomial's top variable using the given scalar.
/// This operation modifies the polynomial in-place.
Not: "The top variable of the polynomial is bound using..."

What Gets Explained vs What Doesn't
Always explained:

Non-obvious parameter roles: * claim - The claimed sum that the prover asserts
Why a constraint is skipped: // we do not need to check if poly(0) + poly(1) = e, as decompress() already ensures that holds
Performance rationale: // This saves N-1 Montgomery reductions per round.
Capacity invariants: // With a u64 limb, we can accumulate up to 2^64 products before overflow.
Never explained:

Trivial operations (push, len, clone)
What the code obviously does by reading it
Business/product context
Error Variant Docs
Uniform formula: /// returned if/when <condition> — always lowercase after "returned", always a conditional phrase:


/// returned if the supplied witness is not of the right length
/// returned when the transcript engine encounters an overflow
/// returned when attempting to divide by zero
Inline Comments
Short, present-tense label then colon, or just a phrase. No full stops at end:


// verify that there is a univariate polynomial for each round
// append the prover's message to the transcript
// derive the verifier's challenge for the next round
// evaluate the claimed degree-ell polynomial at r_i
Multi-line inline comments use full sentences only when explaining a non-obvious invariant:


// Overflow here means we've exceeded the accumulator's capacity (~2^64 products)
// - this should never happen in valid usage since sumcheck polynomials are bounded
// by practical sizes.
Math in Prose
Math identifiers written in $...$ even mid-sentence, no verbosity around them:


/// A multilinear extension of a polynomial $Z(\cdot)$, denote it as $\tilde{Z}(x_1, ..., x_m)$
/// Vector $Z$ indicates $Z(e)$ where $e$ ranges from $0$ to $2^m-1$.
Not: "The vector named Z stores the evaluations of the function Z of e..."

Key Rules Distilled
Start doc comments with a verb (Verifies, Fold, Creates, Returns, Binds) — never with "This"
One sentence per line in multi-line docs; no run-ons
Inline comments are labels, not narration — describe what, occasionally why, never how
Error docs: returned if/when <lowercase condition> — uniform, terse
No trailing punctuation on inline comments; period on doc comment summary lines only
Math identifiers stay in LaTeX even in plain English sentences
Rationale comments earn their place — only appear when the code would otherwise look wrong or surprising
